## 规划中的技术栈与架构

以下设计来自 `notes/01`，是未来实现的理论参考，**尚不存在对应代码**。命名（如 `Mprpc*`）沿用参考项目，正式实现时可能变化。

三大技术支柱：

| 组件 | 角色 | 核心能力 |
|------|------|----------|
| `ZooKeeper` | 服务注册与发现中心 | 服务名 = 永久性节点（Persistent Node），方法名 = 临时性节点（Ephemeral Node），节点数据 = IP:Port |
| `ProtoBuf` | 序列化/反序列化 + 接口定义 | `.proto` 经 `protoc` 生成 Stub 类（Caller 用）与 Service 基类（Callee 继承重写） |
| `Muduo` | 网络通信库 | 基于 Multi-Reactor 模型的多线程网络库 |

规划中的核心组件（详见 `notes/01` 知识点6~13）：

- `MprpcApplication::Init(argc, argv)` —— 框架初始化（读配置文件）
- `RpcProvider`（服务端）—— `NotifyService()` 通过 protobuf 反射注册服务到两级映射表（服务名 → 方法描述符），`Run()` 完成"读配置 → 建 TcpServer → 设线程 → 连 ZK → 挂节点 → 进事件循环"六步
- `MprpcChannel`（客户端）—— 继承 `google::protobuf::RpcChannel` 重写 `CallMethod()`，六步调用链：组装标识 → 序列化 → 查 ZooKeeper → 建 TCP 连接 → 发送 → 阻塞等待响应
- `ZkClient` —— ZooKeeper 客户端封装；`zookeeper_init` 是异步的，用信号量 + Watcher 回调把 `Start()` 同步化
- 异步日志模块 —— 消息队列 + 后台写盘线程（生产者-消费者模型）

已知的设计取舍：当前参考实现为**短连接**（服务端发送响应后主动断开）与**同步阻塞调用**。

1. **README.md 严格面向读者**：只写什么是 RPC、为什么需要、应用场景、Roadmap、笔记链接。实现细节（调用流程图、组件内部、技术表格、代码）**不得写入 README**，它们属于学习笔记。这是用户的明确规则。
2. **notes/ 目录**存放编号学习笔记（`01-`、`02-`……），通过学习材料（文章、截图、视频）生成，格式约定：
   - 顶部标注来源（链接 + 作者）
   - 按原始材料顺序组织为 10~15 个带锚点的知识点（`知识点N`）
   - 流程用状态机式列表（状态/事件/动作/下一状态）描述
   - 术语采用中英双语 `中文(English)` 写法
   - 图片存放在 `notes/images/`
3. **工作流**：在生成或重构内容之前，先给出要点大纲（要点）供用户审阅，**获得批准后再执行**。
4. **保留手工修改**：用户经常在生成后手动编辑文件——再次修改这些文件时，务必保留用户的手工改动。
